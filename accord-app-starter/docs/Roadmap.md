# Accord Roadmap

## Phase 0 — Foundation and validation

- Confirm Utah MVP workflows, licensed form rights, and brokerage review requirements.
- Finalize domain vocabulary, tenancy, approval state machine, and security model.
- Validate the transaction schema against representative Utah deals.
- Prototype extraction and form mapping with test data.
- Establish product analytics and AI quality evaluation without capturing sensitive content.

Exit: signed-off MVP scope, compliant form source, test corpus, and architecture decisions.

## Phase 1 — Utah web MVP

- Authentication and agent/team/brokerage account setup.
- Office Profile defaults and playbook.
- Deal Desk and transaction workspace.
- Transcript paste/manual intake; consent-gated recording only if production-ready.
- Structured extraction with sources, confidence, and conflict handling.
- Utah form recommendation and Settings-managed Contract Library.
- Agent approval workflow and versioned package generation.
- Accord Cloud storage, audit events, retention basics, export, and deletion.

Exit: an agent can create, review, approve, generate, and retrieve a Utah draft package with a complete audit trail.

## Phase 2 — Team and brokerage operations

- Delegated transaction coordinator workflow.
- Brokerage policies, required review paths, and organization templates.
- Broker review queue, exception reporting, and operational dashboards.
- Stronger lifecycle management, legal holds, and administrative exports.
- E-signature and transaction-system integrations selected through validation.

## Phase 3 — Connected storage and intelligence

- Optional OneDrive connection, folder mapping, sync status, conflict recovery, and disconnect behavior.
- Conversation capture improvements and provider resilience.
- Deadline and obligation intelligence with source traceability.
- Reusable office learning based on explicitly approved rules—not silent model memory.

## Phase 4 — Expansion

- Additional states only after form licensing, legal review, mappings, and evaluations.
- Broader transaction types and ecosystem integrations.
- Mobile-native experience if usage evidence warrants it.
- Infrastructure migration or service extraction only when operational evidence justifies it.

## Prioritization rule

Trust and correctness outrank breadth. A feature is not ready because a demo works; it is ready when permissions, auditability, failure recovery, accessibility, privacy, and support are defined.
